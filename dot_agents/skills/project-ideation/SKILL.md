# Project Ideation

## 基本方針

- 目的を「作って面白いアイデアの具体化」に限定する。
- 収益化・事業化・市場規模・売上見込みは扱わない。
- 実装詳細（テーブル設計、API仕様、アーキテクチャ、ライブラリ比較、コード構成）に入らない。
- ユーザー指定の技術やライブラリがある場合のみ、実装議論ではなく制約として扱う。
- 出力は必ず `IDEA.md` にまとめる。

## ワークフロー

1. スコープを固定する。
2. 不明点を質問で潰す。
3. 複数サブエージェントで並列検討する。
4. 意見を統合して改善ループを回す。
5. `IDEA.md` を出力する。

## 1) スコープ固定

最初に以下を明文化する。

- このセッションの対象: `何を作るか / なぜ面白いか / どの体験を核にするか`
- このセッションの非対象: `どう実装するか`
- 技術制約: ユーザー明示がある場合のみ固定
- 事業性検討: 対象外

## 2) 不明点の解消

不明点がある限り、質問してから先に進む。

- Plan mode で `request_user_input` が使える場合:
  - 1回の呼び出しは最大3問までにしつつ、必要な回数だけ繰り返して質問する。
  - 推奨選択肢を先頭に置く。
- `request_user_input` が使えない場合:
  - 通常メッセージで必要なだけ質問する。
  - 回答負荷が高い場合は複数回に分割して聞く。

質問の優先順:

1. 一番「ニヤつく瞬間」は何か
2. 何が面白いと感じるのか（自分向けの動機でよい）
3. やらないことは何か
4. 成功/失敗をどう判定するか
5. 固定したい技術制約はあるか

必要なら追加で聞く観点（任意）:

- 体験の具体像
- 既存との差分
- 公開範囲や時間制約

## 3) サブエージェント並列検討

必ず複数ロールで並列に検討する。各ロールには担当観点だけを渡し、実装詳細は対象外と明示する。

推奨ロール（4つ）:

1. 体験設計ロール
2. 新奇性ロール
3. 制約整合ロール
4. 反証ロール

ロール依頼テンプレート（短文）:

- 体験設計: 「核となる体験と、作るとニヤつく瞬間を1文で示せ」
- 新奇性: 「面白さのフックを1-2点で示せ（差分は任意）」
- 制約整合: 「ユーザー指定技術があれば制約として反映し、矛盾を指摘せよ」
- 反証: 「面白さが弱い点を指摘し、改善案を1つだけ提案せよ」

## 4) 統合ルール

サブエージェントの結果は次の順で統合する。

1. 必須要件充足:

- スコープ固定に違反していないか
- ユーザー指定制約を満たしているか

2. 評価軸スコア（各0-3点）:

- 作る動機
- 驚き・新奇性
- 遊び心・実験性
- 最初の達成の近さ
- 学びの密度
- 語りたさ

3. 衝突解消:

- 優先順位: 必須要件 > 体験価値 > 複雑さの低さ > 新奇性
- 同点なら「作る動機が最も明確な案」を採用
- 未解決対立は追加質問1つで解消する

4. 改善ループ:

- 最低点の軸を1つ選ぶ
- その軸だけを上げる変更案を1つ追加
- 再評価し、最低点2以上を目標に1-2周だけ回す

## 5) ガードレール

- 禁止語トリガー（例: DB、ER図、APIエンドポイント、クラス設計、フレームワーク比較）が出たら、アイデア層に戻す。
- 技術名は「制約欄」にのみ記録し、実装議論はしない。
- 提案は1案あたり最大5行で簡潔に保つ。

## 6) 最終出力: IDEA.md

最終的に以下の形式で `IDEA.md` を作成する。

```markdown
# IDEA

## 1. 一言コンセプト

-

## 2. 作って面白い理由（作る動機）

-

## 3. コア体験（任意）

-

## 4. 差分（既存との違い）

- 任意

## 5. 制約

- ユーザー指定の技術・ライブラリ:
- その他制約:

## 6. 非目標（やらないこと）

-
-

## 7. 成功条件 / 失敗条件

- 成功:
- 失敗:

## 8. 統合根拠

- 採用した意見:
- 棄却した意見と理由:
- スコア要約:

## 9. 未解決の問い

-
```

必要なら最後に「次回深掘りする問い」を1-3個だけ添える。
